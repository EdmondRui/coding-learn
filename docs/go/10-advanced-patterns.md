## 10.1 依赖注入（Dependency Injection）

### 10.1.1 为什么需要 DI

在没有 DI 的代码中，组件直接创建依赖，导致紧耦合、难以测试：

```go
// 紧耦合：UserService 直接创建数据库连接
type UserService struct{}
func (s *UserService) GetUser(id int) User {
    db := &sql.DB{}  // 硬编码依赖
    // ...
}
```

DI 的核心思想是：**依赖由外部注入，而非内部创建**。

### 10.1.2 手动构造函数注入

Go 最推崇的 DI 方式——简单、显式、无魔法：

```go
type Logger interface {
    Log(string)
}

type ConsoleLogger struct{}
func (c ConsoleLogger) Log(msg string) { fmt.Println(msg) }

type UserService struct {
    logger Logger
    db     *sql.DB
}

// 构造函数注入——所有依赖一目了然
func NewUserService(logger Logger, db *sql.DB) *UserService {
    return &UserService{logger: logger, db: db}
}

// 使用方负责组装
func main() {
    logger := ConsoleLogger{}
    db, _ := sql.Open("postgres", "dsn")
    svc := NewUserService(logger, db)
    svc.logger.Log("service created")
}
```

### 10.1.3 接口注入（更灵活）

将注入逻辑提取为接口，适合框架层：

```go
type Configurable interface {
    Configure(*Config)
}

type Server struct {
    port int
}

func (s *Server) Configure(c *Config) {
    s.port = c.Port
}

func Init(configs ...Configurable) *Config {
    cfg := &Config{Port: 8080} // 默认值
    for _, c := range configs {
        c.Configure(cfg)
    }
    return cfg
}
```

### 10.1.4 Wire：编译期 DI 框架

[Wire](https://github.com/google/wire) 是 Google 出品的代码生成工具，在编译期生成依赖组装代码，**不依赖运行时反射**：

```go
// wire.go（开发者编写）
//go:build wireinject
// +build wireinject

package main

import "github.com/google/wire"

func InitializeApp() (*App, error) {
    wire.Build(NewConfig, NewDB, NewLogger, NewUserService, NewApp)
    return nil, nil // 占位，实际由 wire 生成
}
```

```go
// wire_gen.go（wire 自动生成，无需手写）
// Code generated by Wire. DO NOT EDIT.

package main

func InitializeApp() (*App, error) {
    config := NewConfig()
    db, err := NewDB(config)
    if err != nil {
        return nil, err
    }
    logger := NewLogger(config)
    userService := NewUserService(db, logger)
    app := NewApp(userService)
    return app, nil
}
```

**Wire 的优势**：依赖图在编译期校验，无反射开销，生成的代码可读、可调试。

### 10.1.5 fx：运行时 DI 框架

Uber 的 [fx](https://github.com/uber-go/fx) 基于反射，提供生命周期管理：

```go
import "go.uber.org/fx"

func main() {
    fx.New(
        fx.Provide(
            NewConfig,
            NewDB,
            NewLogger,
            NewUserService,
            NewHTTPServer,
        ),
        fx.Invoke(func(srv *http.Server) {
            go srv.ListenAndServe()
        }),
    ).Run()
}
```

| 方式 | 优点 | 缺点 |
|------|------|------|
| 手动 DI | 简单、透明、无依赖 | 大型项目 boilerplate 多 |
| Wire | 编译期检查、无反射 | 需额外工具链 |
| fx | 功能丰富、生命周期管理 | 反射开销、错误在运行时暴露 |

**建议**：中小项目用手动 DI；大项目用 Wire；微服务场景考虑 fx。

## 10.2 反射的应用

### 10.2.1 reflect 包基础

Go 的反射基于两大核心类型：`reflect.Type` 和 `reflect.Value`。

```go
import "reflect"

func Inspect(v any) {
    t := reflect.TypeOf(v)   // 获取类型信息
    val := reflect.ValueOf(v) // 获取值信息

    fmt.Println("类型:", t.Name())
    fmt.Println("种类:", t.Kind())  // Kind 是底层种类（struct, slice, ptr...）
    fmt.Println("值:", val.Interface())

    // 通过反射修改变量（必须传指针）
    ptr := reflect.ValueOf(&v)
    if ptr.Elem().CanSet() {
        ptr.Elem().SetInt(42)
    }
}
```

### 10.2.2 结构体标签读取

反射最广泛的用途之一——读取 struct tag：

```go
type User struct {
    Name  string `json:"name"  validate:"required,max=100"`
    Email string `json:"email" validate:"required,email"`
    Age   int    `json:"age"   validate:"min=0,max=150"`
}

func ReadTags(v any) {
    t := reflect.TypeOf(v)
    for i := 0; i < t.NumField(); i++ {
        field := t.Field(i)
        jsonTag := field.Tag.Get("json")
        validateTag := field.Tag.Get("validate")
        fmt.Printf("字段 %s → json:%s  validate:%s\n",
            field.Name, jsonTag, validateTag)
    }
}

// 输出:
// 字段 Name → json:name  validate:required,max=100
// 字段 Email → json:email  validate:required,email
// 字段 Age → json:age  validate:min=0,max=150
```

### 10.2.3 简单验证器（利用反射）

```go
func Validate(v any) error {
    val := reflect.ValueOf(v)
    t := val.Type()
    for i := 0; i < t.NumField(); i++ {
        field := t.Field(i)
        tag := field.Tag.Get("validate")
        if tag == "" {
            continue
        }
        fv := val.Field(i)
        // 解析 tag：required
        if tag == "required" && fv.String() == "" {
            return fmt.Errorf("field %s is required", field.Name)
        }
    }
    return nil
}
```

标准库 `encoding/json`、`encoding/xml`、`gorm`、`validator` 等均依赖反射读取标签。

### 10.2.4 反射的性能代价

```go
func BenchmarkDirect(b *testing.B) {
    u := User{Name: "alice"}
    for i := 0; i < b.N; i++ {
        _ = u.Name
    }
}

func BenchmarkReflect(b *testing.B) {
    u := User{Name: "alice"}
    for i := 0; i < b.N; i++ {
        v := reflect.ValueOf(u)
        _ = v.FieldByName("Name").String()
    }
}

// 典型结果：反射慢 2~3 个数量级
```

**替代方案**：
- 代码生成（Wire、`stringer`、`protoc-gen-go`）——编译期完成元编程
- 类型断言 + 接口 —— 避免反射的动态分发
- `unsafe` —— 极端场景下替代反射访问字段（慎重）

## 10.3 unsafe 包

`unsafe` 包绕过 Go 类型安全，操作内存布局。**非必要不使用，但了解它有助于理解 Go 底层**。

### 10.3.1 unsafe.Pointer

`unsafe.Pointer` 是通用指针类型，可做四种转换的桥梁：

```
T 指针 ←→ unsafe.Pointer ←→ uintptr（算术运算）
```

```go
import "unsafe"

func Float64ToBits(f float64) uint64 {
    // 安全等价：math.Float64bits(f)
    return *(*uint64)(unsafe.Pointer(&f))
}

type H struct {
    A int16  // 2 字节
    B int32  // 4 字节（偏移 4，有对齐填充）
    C int8   // 1 字节
}

func Offset() {
    h := H{A: 1, B: 2, C: 3}
    fmt.Println(unsafe.Offsetof(h.B)) // 输出: 4（而非 2）
    fmt.Println(unsafe.Sizeof(h))     // 输出: 12（含填充字节）
}
```

### 10.3.2 字符串与字节切片零拷贝转换

这是 Go 社区广为流传的性能技巧：

```go
import "unsafe"

// StringToBytes 零拷贝将 string 转 []byte
func StringToBytes(s string) []byte {
    return unsafe.Slice(unsafe.StringData(s), len(s))
}

// BytesToString 零拷贝将 []byte 转 string
func BytesToString(b []byte) string {
    return unsafe.String(unsafe.SliceData(b), len(b))
}

// 注意：转换后的 []byte 不可修改！修改会导致未定义行为
```

Go 1.20 引入的 `unsafe.String`、`unsafe.StringData`、`unsafe.Slice`、`unsafe.SliceData` 让这类转换更安全（但仍需遵守只读约束）。

### 10.3.3 典型使用场景

| 场景 | 说明 |
|------|------|
| 系统调用 | 将 Go 切片指针传给 C 的 `read`/`write` |
| 内存池 | 操作预分配的大块内存 |
| 序列化 | 零拷贝编解码（如 flatbuffers） |
| 性能热点 | 避免反射的字段访问 |

### 10.3.4 风险

- **无类型安全**：指针运算越界导致 coredump
- **GC 不感知**：`uintptr` 不被 GC 追踪，对象被回收后 dangling pointer
- **可移植性**：不同架构的对齐方式和大小端不同
- **兼容性**：Go 版本升级可能改变内存布局

> **铁律**：在 `import "unsafe"` 的注释中写明为什么必须用它，以及如何验证正确性。

## 10.4 CGo

### 10.4.1 Go 调用 C 函数

```go
package main

/*
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

// C 函数：将字符串转为大写（原地）
void to_upper(char *s, int len) {
    for (int i = 0; i < len; i++) {
        if (s[i] >= 'a' && s[i] <= 'z') {
            s[i] -= 32;
        }
    }
}

// C 函数：返回 hello 字符串
char *greet(const char *name) {
    char *buf = malloc(256);
    snprintf(buf, 256, "Hello, %s!", name);
    return buf;
}

// C 函数：释放内存
void free_str(void *p) { free(p); }
*/
import "C"
import (
    "fmt"
    "unsafe"
)

func main() {
    // 1. 调用 C 函数操作字节数组
    buf := []byte{'h', 'e', 'l', 'l', 'o'}
    C.to_upper((*C.char)(unsafe.Pointer(&buf[0])), C.int(len(buf)))
    fmt.Println(string(buf)) // HELLO

    // 2. 调用 C 函数返回字符串
    cname := C.CString("Go")
    defer C.free(unsafe.Pointer(cname)) // C 堆内存需手动释放

    result := C.greet(cname)
    defer C.free_str(unsafe.Pointer(result))
    fmt.Println(C.GoString(result)) // Hello, Go!
}
```

### 10.4.2 C 调用 Go 函数（跨语言回调）

通过 `//export` 指令导出 Go 函数给 C 调用：

```go
package main

/*
#include <stdio.h>

// C 函数指针类型
typedef void (*callback_fn)(int, const char*);

// C 函数：接收回调并在内部调用
void run_callback(callback_fn cb) {
    cb(42, "from C");
}
*/
import "C"
import "fmt"

//export GoCallback
func GoCallback(id C.int, msg *C.char) {
    fmt.Printf("C called Go: id=%d, msg=%s\n", id, C.GoString(msg))
}

func main() {
    C.run_callback((C.callback_fn)(C.GoCallback))
    // 输出: C called Go: id=42, msg=from C
}
```

### 10.4.3 CGo 的性能开销

```go
// 对比：纯 Go 调用 vs CGo 调用
func BenchmarkPureGo(b *testing.B) {
    for i := 0; i < b.N; i++ {
        _ = 1 + 1
    }
}

// CGo 调用有 ~50ns 固定开销（栈切换、调度阻塞）
```

**CGo 调用流程**：
1. 切换 Go 栈 → 系统栈（~20-50ns）
2. C 代码执行
3. 切换回 Go 栈
4. M 可能被 P 剥离（若 C 调用阻塞）

### 10.4.4 该用 CGo 的场景

| ✅ 应该用 | ❌ 不该用 |
|-----------|-----------|
| 调用无可替代的 C 库（SQLite、OpenSSL、FFmpeg） | 有纯 Go 实现替代 |
| 系统 API 无 Go 绑定（Windows API 某些场景） | 性能关键路径上的小函数调用 |
| 复用千万行 C/C++ 代码库 | 仅仅是图方便不用重写 |

> **经验法则**：CGo 是最后的手段。优先找纯 Go 库；其次用 `os/exec` 调用外部进程；最后才考虑 CGo。

## 10.5 插件系统

### 10.5.1 plugin 包（Go 原生插件）

`plugin` 包支持在运行时加载 `.so` 共享库（仅 Linux/macOS，**不支持 Windows**）：

```go
// ----- plugin/greeter.go -----
package main

import "fmt"

var Greeter = greeter{}

type greeter struct{}

func (g greeter) Greet(name string) string {
    return fmt.Sprintf("Hello, %s!", name)
}

// ----- main.go -----
package main

import "plugin"

func main() {
    p, err := plugin.Open("./greeter.so")
    if err != nil {
        panic(err)
    }
    sym, err := p.Lookup("Greeter")
    if err != nil {
        panic(err)
    }
    greeter := sym.(struct{ Greet(string) string })
    fmt.Println(greeter.Greet("Go Plugin")) // Hello, Go Plugin!
}
```

**构建**：`go build -buildmode=plugin -o greeter.so plugin/greeter.go`

**限制**：
- 插件与应用必须使用**完全相同的 Go 版本**编译
- 所有依赖的版本也必须一致
- 无法热更新（卸载后无法重新加载）
- Windows 不支持

### 10.5.2 基于接口的插件模式

更灵活的方式——通过接口约定插件契约：

```go
// sdk/plugin.go — 插件 SDK
package sdk

type Plugin interface {
    Name() string
    Process(input string) (string, error)
}

type BasePlugin struct{}
func (b BasePlugin) Name() string { return "base" }
```

```go
// plugins/upper/main.go — 插件实现
package main

import "sdk"

type UpperPlugin struct{ sdk.BasePlugin }
func (u UpperPlugin) Process(input string) (string, error) {
    return strings.ToUpper(input), nil
}

var Plugin UpperPlugin // 插件入口符号
```

```go
// main.go — 插件加载器
func LoadPlugins(dir string) []sdk.Plugin {
    entries, _ := os.ReadDir(dir)
    var plugins []sdk.Plugin
    for _, e := range entries {
        if !strings.HasSuffix(e.Name(), ".so") {
            continue
        }
        p, err := plugin.Open(filepath.Join(dir, e.Name()))
        if err != nil {
            continue
        }
        sym, err := p.Lookup("Plugin")
        if err != nil {
            continue
        }
        plug, ok := sym.(sdk.Plugin) // 类型断言到接口
        if ok {
            plugins = append(plugins, plug)
        }
    }
    return plugins
}
```

### 10.5.3 配置驱动插件

将插件注册信息放在配置文件中，避免硬编码：

```yaml
# plugins.yaml
plugins:
  - name: upper
    path: ./plugins/upper.so
    enabled: true
  - name: reverse
    path: ./plugins/reverse.so
    enabled: false
```

通过配置决定加载哪些插件，实现真正的可插拔架构。

## 10.6 Go 与 WebAssembly

### 10.6.1 快速开始

Go 支持将代码编译为 WASM，运行时需 JavaScript 侧的 "胶水代码"：

```bash
GOOS=js GOARCH=wasm go build -o main.wasm main.go
# 将 Go 运行时的 JS 支持文件拷贝到静态目录
cp "$(go env GOROOT)/misc/wasm/wasm_exec.js" .
```

```go
// main.go
package main

import "syscall/js"

func main() {
    // 注册 Go 函数到 JavaScript 全局
    js.Global().Set("add", js.FuncOf(func(this js.Value, args []js.Value) any {
        a := args[0].Int()
        b := args[1].Int()
        return a + b
    }))

    // 保持程序运行（防止 main 退出）
    select {}
}
```

```html
<!-- index.html -->
<script src="wasm_exec.js"></script>
<script>
  const go = new Go();
  WebAssembly.instantiateStreaming(fetch("main.wasm"), go.importObject)
    .then(result => {
      go.run(result.instance);
      console.log(add(3, 4)); // 7
    });
</script>
```

### 10.6.2 DOM 操作

```go
func ManipulateDOM() {
    doc := js.Global().Get("document")
    body := doc.Get("body")

    // 创建元素
    div := doc.Call("createElement", "div")
    div.Set("innerHTML", "<h1>Hello from Go WASM!</h1>")
    div.Get("style").Set("color", "blue")
    body.Call("appendChild", div)

    // 事件监听
    btn := doc.Call("createElement", "button")
    btn.Set("textContent", "Click me")
    btn.Call("addEventListener", "click", js.FuncOf(func(this js.Value, args []js.Value) any {
        js.Global().Call("alert", "Go WASM button clicked!")
        return nil
    }))
    body.Call("appendChild", btn)
}
```

### 10.6.3 Go ↔ JavaScript 互调

```go
// Go 调用 JS
result := js.Global().Call("JSON.parse", `{"a":1}`)
fmt.Println(result.Get("a").Int()) // 1

// JS 回调 Go（注册函数）
callback := js.FuncOf(func(this js.Value, args []js.Value) any {
    data := args[0].String()
    println("JS sent:", data)
    return "ack"
})
js.Global().Set("onGoData", callback)
```

### 10.6.4 适用场景

| 推荐使用 | 不推荐使用 |
|----------|-----------|
| 移植现有 Go 逻辑到浏览器 | 重 DOM/UI 交互的应用 |
| 计算密集型任务（图像处理、数据压缩） | 启动时间敏感场景（WASM 加载慢） |
| 跨平台共享业务逻辑 | 需大量 JS 生态库的场景 |
| 边缘计算（Cloudflare Workers 等） | 简单页面——直接用 JS 更省事 |

**注意**：Go WASM 二进制体积较大（至少 ~2MB，含运行时）。用 `-ldflags="-s -w"` 缩小体积，配合 gzip 压缩传输。

## 10.7 实战：构建一个完整的 CLI 工具

本节使用 [cobra](https://github.com/spf13/cobra) 构建一个文件检索工具 `fsearch`，具备子命令、配置文件和优雅退出。

### 10.7.1 项目结构

```
fsearch/
├── main.go
├── cmd/
│   ├── root.go       # 根命令
│   ├── search.go     # search 子命令
│   └── config.go     # config 子命令
├── config/
│   └── config.go     # 配置读取
└── search/
    └── search.go     # 核心逻辑
```

### 10.7.2 根命令与配置文件

```go
// cmd/root.go
package cmd

import (
    "fmt"
    "os"
    "os/signal"
    "syscall"

    "github.com/spf13/cobra"
    "fsearch/config"
)

var cfgFile string
var cfg *config.Config

var rootCmd = &cobra.Command{
    Use:   "fsearch",
    Short: "文件检索工具",
    Long:  `fsearch 是一个支持正则匹配、递归搜索的文件查找工具。`,
    PersistentPreRunE: func(cmd *cobra.Command, args []string) error {
        var err error
        cfg, err = config.Load(cfgFile)
        return err
    },
    Run: func(cmd *cobra.Command, args []string) {
        fmt.Println("fsearch — 使用 -h 查看帮助")
    },
}

func Execute() {
    rootCmd.PersistentFlags().StringVarP(
        &cfgFile, "config", "c", "config.yaml", "配置文件路径",
    )

    // 信号处理 + 优雅退出
    ctx, stop := signal.NotifyContext(
        context.Background(),
        syscall.SIGINT, syscall.SIGTERM,
    )
    defer stop()

    go func() {
        <-ctx.Done()
        fmt.Println("\n收到退出信号，正在清理...")
        // 关闭文件句柄、数据库连接等
        os.Exit(0)
    }()

    if err := rootCmd.ExecuteContext(ctx); err != nil {
        fmt.Fprintln(os.Stderr, err)
        os.Exit(1)
    }
}
```

### 10.7.3 search 子命令

```go
// cmd/search.go
package cmd

import (
    "fmt"
    "path/filepath"

    "github.com/spf13/cobra"
)

var searchCmd = &cobra.Command{
    Use:   "search [path] [pattern]",
    Short: "递归搜索匹配的文件",
    Args:  cobra.ExactArgs(2),
    RunE: func(cmd *cobra.Command, args []string) error {
        root := args[0]
        pattern := args[1]

        // 利用 cobra 的上下文做优雅退出
        ctx := cmd.Context()

        matches, err := filepath.Glob(filepath.Join(root, pattern))
        if err != nil {
            return fmt.Errorf("搜索失败: %w", err)
        }

        for _, m := range matches {
            select {
            case <-ctx.Done():
                fmt.Println("\n搜索已被中断")
                return nil
            default:
                fmt.Println(m)
            }
        }
        return nil
    },
}

func init() {
    rootCmd.AddCommand(searchCmd)
    searchCmd.Flags().BoolP("regex", "r", false, "使用正则匹配")
}
```

### 10.7.4 配置模块

```go
// config/config.go
package config

import (
    "os"

    "gopkg.in/yaml.v3"
)

type Config struct {
    DefaultPath string   `yaml:"default_path"`
    Extensions  []string `yaml:"extensions"`
    MaxDepth    int      `yaml:"max_depth"`
}

func Load(path string) (*Config, error) {
    data, err := os.ReadFile(path)
    if err != nil {
        // 配置文件不存在时返回默认配置
        if os.IsNotExist(err) {
            return Default(), nil
        }
        return nil, err
    }
    cfg := Default()
    if err := yaml.Unmarshal(data, cfg); err != nil {
        return nil, err
    }
    return cfg, nil
}

func Default() *Config {
    return &Config{
        DefaultPath: ".",
        MaxDepth:    3,
    }
}
```

### 10.7.5 优雅退出详解

```go
// 核心模式：signal.NotifyContext
ctx, stop := signal.NotifyContext(
    context.Background(),
    syscall.SIGINT,  // Ctrl+C
    syscall.SIGTERM, // kill
)
defer stop()

// ctx.Done() 在收到信号后被关闭
// 所有子命令通过 cmd.Context() 获取此 ctx
// 在长时间操作中检查 ctx.Done() 实现可取消操作

// 适用场景：
// - HTTP 服务器 Shutdown
// - 关闭数据库连接池
// - 等待正在运行的 goroutine 完成
// - 清理临时文件
```

**完整的生命周期**：

```
启动 → 解析配置 → 注册子命令 → 运行命令 → 信号到达 →
执行清理 → 优雅退出（≤ 指定超时）→ 进程结束
```

## 总结

本章覆盖了 Go 生态中"高阶但不冷门"的核心主题：

| 主题 | 核心要点 |
|------|---------|
| 依赖注入 | 手动 DI 最推荐；Wire 编译期生成；fx 运行时管理 |
| 反射 | 灵活但有性能代价；代码生成是更好的替代方案 |
| unsafe | 零拷贝与内存操作的利器，必须附带安全性论证 |
| CGo | 最后手段的互操作方案，有显著性能开销 |
| 插件系统 | plugin 包限制较多，接口 + 配置更实用 |
| WebAssembly | 适合计算密集的业务逻辑移植，不适合 UI 场景 |
| CLI 实战 | cobra 组织命令，signal.NotifyContext 实现优雅退出 |

掌握这些模式，你将能应对 Go 工程中的绝大多数"非常规"需求。
